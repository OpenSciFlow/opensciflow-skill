# OpenSciFlow Skill

## Skill Name

OpenSciFlow Skill.

## Purpose

OpenSciFlow Skill teaches agents to inspect verified execution capsules, check environment readiness, select reviewed command templates, run smoke tests when available, execute only after approval, and write run records.

This skill does not promise that scientific tools will run everywhere. It helps an agent make tool requirements, environment assumptions, verification status, run records, and known failures explicit before and after execution.

Core principle:

> OpenSciFlow does not eliminate scientific computing failures. It makes them explicit, checkable, diagnosable, and recordable.

## When To Use

Use this skill when a user asks an agent to:

- inspect a verified execution capsule;
- run or prepare an AI for Science tool;
- check local, GPU, container, or HPC/Slurm readiness;
- choose a reviewed command template from a capsule;
- run a smoke test or example run when evidence exists;
- write or validate a run record;
- decide whether execution must be blocked because metadata or verification evidence is missing.

## When Not To Use

Do not use this skill to:

- make clinical, diagnostic, drug-efficacy, or high-risk scientific claims;
- bypass upstream licenses or citation requirements;
- execute arbitrary shell commands generated by an LLM;
- turn an unverified manifest into a claim of reproducibility;
- replace scientific domain review;
- infer missing capsule fields without marking them as missing;
- claim that computational output is experimental validation.

## Required Inputs

- User request.
- Candidate verified execution capsule, or a clear reason no capsule exists.
- Target environment: local, container, or HPC/Slurm.
- Input file metadata for smoke test, example run, or full execution.
- Safety constraints and allowed run directory.

## Optional Inputs

- Existing `opensciflow.yaml` manifest.
- Available model-weight paths.
- User-provided environment details.
- Hardware and scheduler limits.
- Existing run directory.
- Prior run records or verified environment matrix.
- Known failure records.

## Expected Outputs

- Parsed task.
- Selected capsule, or refusal.
- Readiness checks.
- Environment readiness report.
- Missing requirements.
- Known failure warnings.
- Proposed execution plan.
- Rendered command template, or refusal.
- Approval request if execution is possible.
- Run-record template or final run record.

## Step-by-Step Behavior

1. Parse the user request into domain, task type, inputs, desired outputs, and execution mode.
2. Locate a candidate verified execution capsule.
3. Read the capsule manifest, environment spec, command templates, smoke tests, verified environment matrix, run records, and known failures.
4. Validate required fields without guessing.
5. Check readiness level conservatively.
6. Check required inputs, model weights, environment, hardware, license, citation, scheduler, and file-system constraints.
7. Report known failure cases that overlap with the current environment.
8. Run smoke tests when available and allowed.
9. Render only reviewed command templates declared in the capsule.
10. Record the command source as `capsule-command-template`, `workflow-template`, or `reviewed-wrapper`.
11. Show rendered command, inputs, outputs, risks, known failures, and limitations to the user.
12. Request explicit approval before execution beyond inspection or smoke-test preparation.
13. Execute only through the approved local runner.
14. Capture logs, return code, artifacts, hashes, versions, citations, licenses, hardware, and environment details.
15. Write a run record after every attempted execution.
16. On failure, write a failed run record and do not invent new shell commands for retries.

## Safety Rules

- Never execute arbitrary shell commands generated by the LLM.
- Only render reviewed command templates declared in the capsule.
- Always check environment requirements before execution.
- Always run smoke tests when available.
- Always write a run record.
- Always report known failure cases.
- Always fail closed when required metadata is missing.
- Do not claim reproducibility beyond the verified environment matrix.
- Preserve license and citation metadata in approval summaries and run records.
- Do not write outside approved run directories unless the user explicitly approves a path.
- Do not download model weights without showing source, license, and checksum status.

## Refusal Rules

Refuse execution when:

- no capsule, reviewed manifest, or reviewed workflow template is available;
- capsule or manifest schema validation fails;
- required inputs are missing;
- required model weights are missing;
- command template references undeclared variables;
- wrapper script is requested without review metadata;
- license or citation metadata is missing or unclear;
- the user asks for clinical diagnosis, drug efficacy judgment, or high-risk scientific conclusions;
- the workflow request is outside declared capabilities;
- execution would require arbitrary LLM-generated shell;
- the task writes outside approved run directories;
- the target environment is outside the capsule's verified environment matrix and the user is asking for a reproducibility claim.

If the user explicitly wants an unverified exploratory attempt, the agent may prepare a review-only plan, but it must label the attempt as unverified and still use reviewed command templates.

## How To Read A Capsule

Read fields without guessing:

- capsule identity and declared readiness;
- `opensciflow.yaml` manifest fields;
- inputs, outputs, parameters, and artifacts;
- environment spec, container definition, HPC module requirements, and scheduler assumptions;
- reviewed command templates and allowed placeholders;
- smoke tests and expected minimal outputs;
- run records and their status;
- verified environment matrix;
- known failure records;
- license, citation, safety notes, and limitations.

If a required field is absent, report it as missing. Do not fill it from memory unless the output clearly marks it as an unverified suggestion.

## How To Check Readiness Level

Use readiness levels conservatively:

- `R0`: project indexed only. No execution.
- `R1`: draft manifest. Inspection only.
- `R2`: schema-validated manifest. Inspection and review only.
- `R3`: environment spec and reviewed command templates available. Smoke-test preparation only unless evidence exists.
- `R4`: smoke test passed in one environment. Smoke-test rerun allowed with approval.
- `R5`: example run passed with run record. Example execution allowed with approval and run record.
- `R6`: multi-environment verification. Execution may be planned for matching verified environments.
- `R7`: external reproduction. External reproduction evidence may be cited, but limitations still apply.

R1/R2 can only reduce documentation inspection cost. R4/R5 are the first levels where reduced setup trial-and-error can be discussed cautiously. R6/R7 are the first levels with meaningful cross-environment migration evidence.

Never upgrade a readiness level without evidence.

## How To Render Command Template

Render only placeholders allowed by the capsule:

- `{inputs.<name>}`;
- `{outputs.<name>}`;
- `{parameters.<name>}`;
- `{outputs_dir}`;
- `{run_dir}`;
- `{environment.<name>}` when explicitly declared.

Reject commands containing unreviewed shell composition such as `&&`, `||`, `;`, backticks, `$(`, or multiline shell snippets unless the capsule explicitly declares a reviewed wrapper.

For reviewed wrappers, require:

- wrapper path;
- review status;
- reviewer or reviewing project;
- allowed arguments;
- exact rendered submission or launch command;
- expected run-record fields.

## How To Ask For Approval

Before execution beyond inspection, show:

- selected capsule and declared readiness;
- matching and non-matching verified environments;
- rendered command;
- input paths and expected output paths;
- model-weight source and checksum status;
- license and citation;
- hardware and HPC requirements;
- known failure cases relevant to the current environment;
- limitations and non-claims;
- run directory;
- run-record path.

Ask for explicit approval before running anything beyond inspection or smoke-test preparation.

## How To Write Run Record

Record:

- run id;
- capsule id and manifest id;
- timestamps;
- status and return code;
- command template id and rendered command;
- input and output metadata;
- versions;
- environment;
- model weights;
- hardware;
- logs;
- artifacts;
- file hashes;
- citation;
- license;
- limitations;
- known failure matches;
- warnings.

## How To Handle Failure

Fail closed. Report missing requirements clearly. If execution started, write a run record with `status: failed`, logs, return code, known artifacts, and the suspected failure category.

Do not retry by inventing new shell commands. Retries must use reviewed command templates or reviewed wrapper scripts.

## How To Cite Tools

Carry citation metadata from capsules into:

- approval summary;
- run record;
- generated report;
- final user-facing output.

If citation is unclear, refuse execution or mark the run as blocked before execution.

## How To Report Limitations

Always report limitations from:

- capsule;
- manifest;
- workflow template;
- model weights;
- sample data;
- run environment;
- verified environment matrix;
- failed or skipped steps.

Do not convert model outputs into clinical, binding, drug-efficacy, or experimental claims.
