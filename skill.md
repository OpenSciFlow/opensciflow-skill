# OpenSciFlow Skill

## Skill Name

OpenSciFlow Skill.

## Purpose

Teach AI agents how to use OpenSciFlow manifests, workflow templates, readiness levels, safety rules, and run records for local-first AI for Science tool execution.

The skill reduces context and trial-and-error cost by replacing ad hoc README parsing with structured protocol artifacts.

## When To Use

Use this skill when a user asks an agent to:

- run or prepare an AI for Science tool;
- inspect a scientific tool manifest;
- match a scientific task to a workflow template;
- check local, GPU, or HPC readiness;
- render a reviewed command template;
- produce a reproducible run record;
- decide whether a task must be refused for safety or missing metadata.

## When Not To Use

Do not use this skill to:

- make clinical, diagnostic, drug-efficacy, or high-risk scientific claims;
- bypass upstream licenses or citation requirements;
- execute arbitrary shell generated by an LLM;
- replace scientific domain review;
- infer missing manifest fields without marking them as missing;
- claim that computational output is experimental validation.

## Required Inputs

- User request.
- At least one candidate manifest or a clear reason no manifest exists.
- Candidate workflow template if execution is requested.
- Input file metadata for execution or dry run.
- Target environment: local, container, or HPC/Slurm.
- Safety constraints.

## Optional Inputs

- Available model-weight paths.
- User-provided environment details.
- Hardware and scheduler limits.
- Preferred output directory.
- Existing run directory.
- Prior readiness evidence.

## Expected Outputs

- Parsed task.
- Selected workflow, or refusal.
- Selected manifests, or refusal.
- Readiness checks.
- Missing requirements.
- Proposed execution plan.
- Approval request if execution is possible.
- Warnings and limitations.
- Run-record template or final run record.

## Step-by-Step Behavior

1. Parse the user request into domain, task type, inputs, desired outputs, and execution mode.
2. Match the task to an existing workflow template.
3. Read all referenced `opensciflow.yaml` manifests.
4. Validate manifest and workflow fields.
5. Check readiness level.
6. Check required inputs, model weights, environment, hardware, license, citation, and limitations.
7. Prefer dry-run before full execution.
8. Render only reviewed command templates declared in manifests.
9. Record the command source as `manifest-template`, `workflow-template`, or `reviewed-wrapper`.
10. Show rendered command, inputs, outputs, risks, and limitations to the user.
11. Request approval if execution is possible.
12. Execute only through the local agent or approved runner.
13. Capture logs, return code, artifacts, hashes, versions, citations, licenses, and limitations.
14. Write a run record.
15. On failure, write a failure record when execution has started.

## Safety Rules

- Do not execute arbitrary commands generated by the LLM.
- Execute only reviewed command templates declared in the manifest, workflow template, or reviewed wrapper metadata.
- Always prefer dry-run before real execution.
- Always write a run record.
- Always preserve license and citation metadata.
- Always fail closed when required metadata is missing.
- Do not write outside approved run directories unless the user explicitly approves a path.
- Do not download model weights without showing source, license, and checksum status.

## Refusal Rules

Refuse execution when:

- manifest schema validation fails;
- required inputs are missing;
- required model weights are missing;
- command template references undeclared variables;
- wrapper script is requested without review metadata;
- license or citation metadata is missing or unclear;
- the user asks for clinical diagnosis, drug efficacy judgment, or high-risk scientific conclusions;
- the workflow request is outside declared capabilities;
- execution would require arbitrary LLM-generated shell;
- the task writes outside approved run directories;
- the requested environment cannot be verified enough for the selected readiness level.

## How To Read `opensciflow.yaml`

Read fields without guessing:

- `name`, `version`, `domain`, `task_types`;
- `inputs` and `outputs`;
- `environment`, `hardware`, `model_weights`;
- `execution.local` and optional `execution.slurm`;
- `validation.dry_run_command` and smoke tests;
- `license`, `citation`, `safety_notes`, `limitations`.

If a required field is absent, report it as missing. Do not fill it from memory unless the output clearly marks it as an unverified suggestion.

## How To Select Workflow Template

Prefer an existing workflow template. Check:

- workflow name and task boundary;
- required inputs and outputs;
- steps and DAG;
- required and optional plugins;
- artifact handoff;
- fallback paths;
- reproducibility requirements;
- report template and limitations.

If no workflow matches, propose a review-only draft. Do not execute.

## How To Check Readiness Level

Use readiness levels conservatively:

- `R0`: listed only, no execution.
- `R1`: draft manifest, no execution.
- `R2`: metadata reviewed, no execution by default.
- `R3`: dry-run only.
- `R4`: smoke test only unless user provides explicit inputs.
- `R5`: execution allowed with approval and run record.
- `R6`: workflow execution allowed with workflow approval.

Never upgrade a readiness level without evidence.

## How To Render Command Template

Render only placeholders allowed by the manifest:

- `{inputs.<name>}`;
- `{outputs.<name>}`;
- `{parameters.<name>}`;
- `{outputs_dir}`.

Reject commands containing unreviewed shell composition such as `&&`, `||`, `;`, backticks, `$(`, or multiline shell snippets unless the protocol version explicitly supports a reviewed wrapper.

For reviewed wrappers, require:

- wrapper path;
- review status;
- reviewer or reviewing project;
- allowed arguments;
- exact rendered submission or launch command.

## How To Ask For Approval

Before execution, show:

- selected workflow and manifest;
- readiness level;
- rendered command;
- input paths and expected output paths;
- model-weight source and checksum status;
- license and citation;
- hardware and HPC requirements;
- limitations and non-claims;
- run directory.

Ask for explicit approval before running anything beyond dry-run.

## How To Write Run Record

Record:

- run id;
- workflow id;
- manifest id;
- timestamps;
- status and return code;
- commands;
- input and output metadata;
- versions;
- environment;
- model weights;
- hardware;
- logs;
- artifacts;
- file hashes;
- citation;
- license;
- limitations;
- warnings.

## How To Handle Failure

Fail closed. Report missing requirements clearly. If execution started, write a run record with `status: failed`, logs, return code, and known artifacts.

Do not retry by inventing new shell commands. Retries must use reviewed command templates or reviewed wrapper scripts.

## How To Cite Tools

Carry citation metadata from manifests into:

- approval summary;
- run record;
- generated report;
- final user-facing output.

If citation is unclear, refuse execution or mark the run as blocked before execution.

## How To Report Limitations

Always report limitations from:

- manifest;
- workflow template;
- model weights;
- sample data;
- run environment;
- failed or skipped steps.

Do not convert model outputs into clinical, binding, drug-efficacy, or experimental claims.
